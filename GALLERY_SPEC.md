# Joy's Gallery - 视觉与交互规范 (PRD) v3

## 0. 文档用途

这份文档不再只是“理想方向”，而是当前版本的**执行规范**。  
后续做内容系统、详情页、上传方式时，都以这份文档为准。

文档分为两部分：

1. **已冻结项**：当前已经基本确定，不要轻易改动。
2. **待优化项**：方向明确，但细节还允许继续打磨。

---

## 1. 核心视觉：2.5D 轴测风格 (已冻结)

### 1.1 几何形态

- 放弃真实的 3D 旋转，不做明显透视缩放。
- 卡片的**左右边缘保持垂直**。
- 通过 **Skew Y（负值）** 让上下边缘呈现**左高右低**的倾斜状态。
- 当前实现值：`uSkewY = -0.2`

### 1.2 卡片尺寸

- 卡片整体保持克制，不能铺满页面。
- 当前高度规则：`height = viewport.height * 0.28`
- 宽度根据原始图片比例自动计算：
  - 横图保持横图
  - 竖图保持竖图
  - 不再强制统一横版比例

### 1.3 阵列排布

- 卡片阵列从**左下向右上**延伸。
- 当前间距参数：
  - `x = 1.5`
  - `y = 0.45`
  - `z = -0.05`
- 要求视觉上有层叠关系，但不能过密，也不能太散。

## 2. 材质与光影：玻璃感 (部分冻结)

### 2.1 主卡片材质

- 主卡片使用图片原始颜色，不再额外加统一灰色/黑色滤镜。
- 中心区域必须保持清晰。
- 边缘通过 alpha mask 做渐变淡化，形成“玻璃呼吸感”。
- 当前 shader 的核心思路：
  - `silhouetteAlpha` 负责保持清晰边界
  - `contentAlpha` 负责边缘内容淡化

### 2.2 边缘透明策略

- 当前实现方向是对的：
  - 边界仍然存在
  - 内容边缘轻微淡出
  - 颜色仍然来自图片本身
- 当前参数：
  - `contentAlphaX = smoothstep(0.008, 0.095, edgeDistX)`
  - `contentAlphaY = smoothstep(0.008, 0.09, edgeDistY)`
  - `baseOpacity = 0.62`

### 2.3 厚度感

- 厚度必须非常克制，像**一片薄玻璃 / 薄亚克力**，不能像一块厚黑板。
- 当前实现采用：
  - 顶部面 `topFaceGeometry`
  - 右侧面 `rightFaceGeometry`
- 当前厚度偏移：
  - `x = width * 0.008`
  - `y = height * 0.01`
  - `z = -0.008`
- 当前颜色：
  - 顶部面：`#0d0d0f`
  - 右侧面：`#040405`

## 3. 核心交互：滚轮驱动 (已冻结)

- 进入画廊后，滚轮主要驱动画廊阵列移动。
- 当前交互方式：
  - 使用 `ScrollControls`
  - 整个阵列沿对角线方向平移
- 当前平移逻辑：
  - `targetX = -progress * totalTravelX`
  - `targetY = -progress * totalTravelY`

要求：

- 观感上是顺滑的斜向浏览
- 不出现夸张的“怼脸放大”
- 最前面的卡片自然移出视野

---

## 4. 悬停与点击交互 (部分冻结)

### 4.1 悬停抽出

- 卡片 hover 时向右平滑抽出。
- 不再根据单张图片比例动态计算。
- 当前统一标准值：
  - `hover translateX = 1.25`
- 目标观感：
  - 平均能露出 1/2 以上画面
  - 个别横图/竖图允许有少量误差

### 4.2 跟随鼠标的标题

- hover 时显示标题文本。
- 文本跟随鼠标移动，并带轻微阻尼。
- 文字使用 `mix-blend-difference` 保持对比度。
- 当前实现已可用，但视觉表现还不是最终定稿。

### 4.3 点击详情

- 点击卡片后进入详情页。
- 当前详情页为左右分栏结构：
  - 左侧大图
  - 右侧正文

## 6. 后续功能开发原则

后续新增功能时，优先级如下：

1. 不破坏当前 2.5D 视觉语言
2. 不轻易重写 shader 和厚度结构
3. 优先做内容管理、详情页体验、上传方式
4. 只有在明确需要时，再回来微调厚度与边缘质感

## 7. 内容架构 (已冻结)

### 7.1 顶层频道

- 顶层只保留两个主频道：
  - `Life`
  - `Archive`
- `Life` 是主角，承载个人生活照片与感悟。
- `Archive` 更像收藏展览，承载专辑、歌曲、电影、书籍等文化内容。
- 两个频道是**互斥关系**，一条内容不能同时属于两个频道。

### 7.2 首页结构

- 首页不是纯分流页，而是综合画廊。
- 首页同时展示 `Life` 与 `Archive` 内容。
- 首页保留统一的倾斜卡片通道语言。
- 不强制在卡片上长期显示 `Life / Archive` 标签。
- 频道是结构，图片是感知，hover 信息是辅助。

### 7.3 分类与筛选

- `Archive` 的内部筛选类型先固定为：
  - `Album`
  - `Song`
  - `Film`
  - `Book`
- 先不扩展更多 Archive 子类，后续如确有需要再增加。
- `Life` 暂不做复杂子类，主要通过时间轴组织。

---

## 8. 时间轴设计 (Life 专属)

### 8.1 时间轴角色

- 时间轴不是独立页面主结构，而是附着在当前倾斜卡片通道上的辅助层。
- 它应与卡片通道保持平行关系，作为“轻叙事导航层”存在。
- 滑动时，时间轴同步变化。

### 8.2 时间粒度

- `Life` 的时间轴粒度固定为：`年-月`
- hover 时显示更具体的日期：`年.月.日`
- 时间轴只展示有内容的关键月份，不做高频密集打点。

### 8.3 数据规则

- `timelineMonth` 不需要额外手填。
- `Life` 内容的 `timelineMonth` 直接从 `date` 自动读取年月生成。

---

## 9. Hover 信息规范 (已冻结)

### 9.1 Archive hover

显示两层信息：

1. 类型：`Album / Film / Song / Book`
2. 标题

不显示 caption。

### 9.2 Life hover

显示两层信息：

1. 具体日期：如 `2025.03.09`
2. 标题

不显示 caption。

### 9.3 呈现原则

- hover 信息是辅助识别，不是主视觉。
- 默认状态尽量保持画面纯净。
- hover 信息要轻、准、短。
- 文本继续沿用“跟随鼠标 + 自动反差可读”的策略。

---

## 10. 详情页内容模式 (已冻结)

- 详情页采用**同一骨架，内容自适应**的方式。
- 不做两套割裂的详情页模板。
- 每条内容都必须至少有图片，不允许纯文字条目。
- 允许：
  - 多图 + 长正文
  - 多图 + 短正文
  - 单图 + 一句简短内容
- 列表页 / 卡片页只展示首图。
- 详情页中再展示完整图片组与正文。

---

## 11. 排序规则 (已冻结)

### 11.1 Life

- `Life` 按 `date` 倒序排列。
- 同时按 `date` 生成时间轴中的 `年-月` 节点。

### 11.2 Archive

- `Archive` 按**添加顺序**排序。
- 当前实现上只保留一个 `date` 字段。
- 这个 `date` 的含义定义为：该内容被加入站内记录系统的时间。
- 因此 `Archive` 直接按 `date` 倒序展示。
- 不单独拆分 `addedAt` / `workDate` 字段。

---

## 12. JSON 内容存储方案 (已冻结)

### 12.1 当前方案

先采用最直接的本地文件方案：

- 图片放在：`public/images/...`
- 内容 JSON 放在：`content/...`

### 12.2 建议目录结构

- `content/life/`
- `content/archive/`
- `content/templates/`

### 12.3 模板文件

项目内提供两份模板：

- `content/templates/life-entry.template.json`
- `content/templates/archive-entry.template.json`

后续新增内容时，优先通过复制模板并修改字段完成。

### 12.4 数据字段

#### 公共字段

- `id`
- `channel`
- `title`
- `date`
- `coverImage`
- `images`
- `body`
- `tags`

#### Archive 额外字段

- `archiveType`

#### Life 额外规则

- 不需要手动写 `timelineMonth`
- 系统从 `date` 自动生成
